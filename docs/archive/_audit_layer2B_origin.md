## 3. Analysis of "Problematic" Frequencies

### 3.1 "Cada 2 horas" -- Where is it used? Is it from PDF or seed algorithm?

**Occurrences in the system:**

| Context | Source | Origin | Assessment |
|---------|--------|--------|------------|
| Insert Op 100: Temperature monitoring | Patagonia template + run-seed-complete-inserto.mjs manual items | **CTX/MANUAL** | NOT from any PDF. This is a contextual hardcode by the developer for temperature monitoring at the tapizado operation. The specific 2-hour interval is an engineering judgment for process parameter monitoring. |
| AMFE-derived items with AP=M, severity 7-8 | getCompleteDefaults() algorithm | **ALGO** | NOT from any PDF. This is the algorithm's output for medium-risk items with significant (but not critical) severity. |
| AMFE-derived items with AP=L, severity >= 9 | getCompleteDefaults() algorithm | **ALGO** | NOT from any PDF. Algorithm output for low-AP items with high severity. |
| controlPlanDefaults.ts: AP=L, severity >= 9 | Runtime UI suggestion | **RUNTIME** | NOT from any PDF. Same logic as seed algorithm, applied at runtime. |

**Verdict**: "Cada 2 horas" is NEVER directly from an original PDF. It appears in the original PDFs only as part of the longer string "parada de mas de 1 h" (referring to mechanical interventions, not sampling frequency). As a standalone sampling frequency, it is always an engineering judgment from the seed algorithm or contextual hardcodes.

**Is it reasonable?** YES. For temperature monitoring in tapizado (where vinyl/substrate temperature affects adhesion), a 2-hour monitoring interval is standard practice in automotive trim manufacturing. The AIAG CP 2024 guidelines suggest time-based monitoring for process parameters where AP is medium and the characteristic is not CC.

### 3.3 "Cada turno" -- Where is it used?

**Occurrences in the system:**

| Context | Source | Origin | Assessment |
|---------|--------|--------|------------|
| AMFE-derived items with AP=M, severity < 7 | getCompleteDefaults() algorithm | **ALGO** | NOT from any PDF directly. |
| AMFE-derived items with AP=L, severity 5-8 | getCompleteDefaults() algorithm | **ALGO** | NOT from any PDF directly. |
| controlPlanDefaults.ts: AP=M, severity < 9 | Runtime UI suggestion | **RUNTIME** | Same logic. |
| controlPlanTemplates.ts: Muestreo periodico | Template library | **RUNTIME** | Industry standard. |

**Relationship to PDF "Por turno"**: The original PDFs DO use "Por turno" for packaging/embalaje operations (identification, quantity). The seed scripts use "Cada turno" and "1x turno" as variations. These are semantically identical but represent DIFFERENT contexts:
- PDF "Por turno": packaging operations (labeling, counting) -- EVENT-based, once per shift
- ALGO "Cada turno": reduced monitoring for lower-risk process parameters -- TIME-based sampling

**Verdict**: "Cada turno" has a dual origin:
1. For packaging items: FROM PDF (as "Por turno") -- confirmed match
2. For AMFE-derived process items: ALGO -- algorithm output

**Is it reasonable?** YES. Once-per-shift sampling is the lowest reasonable monitoring frequency in automotive manufacturing. It is used for low-risk items where the process is stable and poka-yoke or other prevention controls exist.

### 3.4 Other Frequencies of Interest

**"Inicio + c/hora" and "Inicio turno + cada hora"**:
- Used for injection weight SPC (Armrest OP 50, Insert OP 60, Top Roll OP 10) and adhesive monitoring (Top Roll OP 20, Telas Termoformadas OP 30)
- Origin: CTX -- all hardcoded in seedApqpDocuments.ts
- Assessment: Standard practice for CC items under SPC. The "inicio" component ensures first-piece approval; the "c/hora" ensures ongoing process stability. This is the standard AIAG recommendation for SPC-controlled injection molding parameters.

**"Inicio + c/50" and "Inicio + c/50 pzas"**:
- Used for troquelado/stamping tool wear monitoring (Telas Planas OP 40, Insert OP 50)
- Origin: CTX -- hardcoded in seedApqpDocuments.ts
- Assessment: Every-50-piece sampling is standard for tooling-related characteristics where progressive wear can cause drift.

**"1x turno"**:
- Used for destructive tests (tensile, peel, weld strength) across all 5 seed products
- Origin: CTX -- hardcoded
- Assessment: Once-per-shift destructive testing is the industry standard. It balances the need for material data against the cost of destroying parts. This matches the original PDF frequency for similar tests.

## 4. Assessment

### 4.1 Were the Original PDFs Correct?

**YES, with high confidence.**

The original PDF Control Plans for Barack Mercosul products were:
- **Reviewed by VWA (Volkswagen Argentina)**: The headrest PDFs are marked as "PdC preliminar" (preliminary Control Plan) for the Patagonia project. The Inserto, Top Roll, and APB PDFs have formal revision levels (Rev 0, Rev 1) and VWA approval fields.
- **Following VDA-MLA structure**: All PDFs use the standard VDA 6.3 / AIAG CP format with process steps, characteristics, specifications, sampling, and reaction plans.
- **Consistent across products**: All 12 PDFs use the same frequency vocabulary ("Por entrega", "Inicio de turno", "Por lote", "Por turno", "Auditoria de Producto") which confirms they were written by the same quality engineering team following a company standard.

**Specific validation points**:

### 4.2 Is the AP+Severity Logic a Valid AIAG-Based Approach?

**YES, with caveats.**

The `getCompleteDefaults()` and `getDefaults()` functions implement an AP-to-sampling mapping that follows the AIAG-VDA FMEA Handbook (2019/2024) principle:

**AIAG principle**: Action Priority (AP) determines the urgency of action. Higher AP requires more intensive controls.

**Implementation in seed scripts**:

| AP | Severity | Frequency | AIAG Alignment |
|---|---|---|---|
| H | any | Cada pieza (100%) | CORRECT -- AP=H means current controls are insufficient; 100% inspection is the maximum level |
| M | >= 9 | Cada hora | REASONABLE -- AP=M with CC-level severity warrants intensive sampling; hourly is standard for SPC |
| M | 7-8 | Cada 2 horas | REASONABLE -- AP=M with SC-level severity; 2-hour interval is common for process parameters |
| M | < 7 | Cada turno | REASONABLE -- AP=M with low severity; shift-level monitoring is appropriate |
| L | >= 9 | Cada 2 horas | REASONABLE -- AP=L means controls are adequate, but high severity warrants monitoring |
| L | 5-8 | Cada turno | REASONABLE -- Low risk, periodic confirmation |
| L | < 5 | Inicio de turno | REASONABLE -- Minimal risk, startup verification only |

**Caveats**:
1. The AIAG-VDA FMEA Handbook does NOT prescribe specific sampling frequencies for each AP level. It says "AP=H requires action" and "AP=M/L may require action based on organization assessment." The specific mapping (AP=H -> 100%, AP=M sev>=9 -> hourly, etc.) is an ENGINEERING JUDGMENT made by the Barack Mercosul quality team.
2. The mapping is a reasonable interpretation but NOT the only valid one. Different OEMs and Tier-1 suppliers use different mappings.
3. The runtime defaults in `controlPlanDefaults.ts` have a SLIGHTLY DIFFERENT mapping than the seed scripts (the seed has an extra severity 7 breakpoint for AP=M that the runtime does not). This is a minor inconsistency documented in Layer 1A.

**Overall verdict**: The algorithmic approach is VALID as a starting point for CP frequency assignment. It correctly applies the AIAG-VDA principle that higher risk requires more intensive controls. The specific numeric breakpoints are reasonable engineering judgments consistent with automotive industry practice.
